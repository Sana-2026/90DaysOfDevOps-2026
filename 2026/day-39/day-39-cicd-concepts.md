### Task 2: CI vs CD

Research and write short definitions (2-3 lines each):

Continuous Integration тАФ what happens, how often, what it catches

Continuous Integration is the practice of **frequently merging code changes into a shared repository**, where automated builds and tests run on every commit or pull request.  
It helps detect **integration issues, build failures, and bugs early** in the development cycle.

**Example:** Developers push code to GitHub and a CI tool like Jenkins automatically runs tests to verify the build.

Continuous Delivery тАФ how it's different from CI, what "delivery" means

Continuous Delivery extends CI by ensuring the application is **always in a deployable state**. After passing tests, the code is automatically prepared for release, but **deployment to production still requires manual approval**.

**Example:** After CI tests pass, the application is packaged and ready to deploy to production with one click.

Continuous Deployment тАФ how it differs from Delivery, when teams use it
Write one real-world example for each.

Continuous Deployment goes one step further than Continuous Delivery by **automatically deploying every change that passes tests directly to production**, without manual approval.

**Example:** A company like Netflix automatically deploys updates to production once automated tests succeed.

### Task 3: Pipeline Anatomy

A pipeline has these parts тАФ write what each one does:

1. Trigger тАФ what starts the pipeline

тЮб The **event that starts the pipeline**, such as a code push, pull request, or scheduled run.

2. Stage тАФ a logical phase (build, test, deploy)

A **logical phase of the pipeline** (e.g., build, test, deploy) that groups related jobs.

3. Job тАФ a unit of work inside a stage

тЮб A **set of tasks executed within a stage** on the same runner.

4. Step тАФ a single command or action inside a job

тЮб A **single command or action inside a job**, such as running tests or installing dependencies.

 
5. Runner тАФ the machine that executes the job
- The **machine or agent that executes the pipeline jobs** (VM, container, or physical machine).

6. Artifact тАФ output produced by a job
тЮб A **file or output generated by a job** (e.g., compiled app, build package) that can be used in later stages.

https://github.com/fastapi/full-stack-fastapi-template/blob/master/.github/workflows/deploy-staging.yml
```
name: Deploy to Staging

on:
  push:
    branches:
      - master

jobs:
  deploy:
    # Do not deploy in the main repository, only in user projects
    if: github.repository_owner != 'fastapi'
    runs-on:
      - self-hosted
      - staging
    env:
      ENVIRONMENT: staging
      DOMAIN: ${{ secrets.DOMAIN_STAGING }}
      STACK_NAME: ${{ secrets.STACK_NAME_STAGING }}
      SECRET_KEY: ${{ secrets.SECRET_KEY }}
      FIRST_SUPERUSER: ${{ secrets.FIRST_SUPERUSER }}
      FIRST_SUPERUSER_PASSWORD: ${{ secrets.FIRST_SUPERUSER_PASSWORD }}
      SMTP_HOST: ${{ secrets.SMTP_HOST }}
      SMTP_USER: ${{ secrets.SMTP_USER }}
      SMTP_PASSWORD: ${{ secrets.SMTP_PASSWORD }}
      EMAILS_FROM_EMAIL: ${{ secrets.EMAILS_FROM_EMAIL }}
      POSTGRES_PASSWORD: ${{ secrets.POSTGRES_PASSWORD }}
      SENTRY_DSN: ${{ secrets.SENTRY_DSN }}
    steps:
      - name: Checkout
        uses: actions/checkout@v6
      - run: docker compose -f compose.yml --project-name ${{ secrets.STACK_NAME_STAGING }} build
      - run: docker compose -f compose.yml --project-name ${{ secrets.STACK_NAME_STAGING }} up -d
```

Trigger

  + Runs on push to master branch
    
Jobs

  + 1 job (deploy)
    
What it does

+ Deploys the app to a staging environment

+ Runs on a self-hosted staging server

+ Uses environment secrets for configuration

+ Builds Docker containers

+ Starts the application with Docker Compose
